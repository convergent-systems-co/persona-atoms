## Test Strategy

`scripts/validate.py` is the TDD gate. It runs red before any implementation file exists and green after all files are present and valid. The script is self-contained (Python stdlib only; no external deps). CI runs it on every PR.

Steps:
1. Write `validate.py` (T1) — confirm it fails on missing files
2. Implement T2–T6
3. Run `validate.py` — confirm 0 exit

## Alternatives Considered

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| Extend starter-persona-atoms schema (simple flat atom) | Less new code | Does not model GOALS.md composition semantics; no voice/role/constraint separation | Rejected — insufficient semantic richness |
| YAML atom files instead of JSON | More human-readable | Inconsistent with prompt-atoms pattern; harder to validate with stdlib | Rejected — JSON aligns with ecosystem standard |
| Single `atom-v1.json` schema discriminated by `type` field | Mirrors prompt-atoms | Conflates 5 distinct vocabularies; makes type-specific required fields impossible | Rejected — each type needs distinct required properties |

## Risk Assessment

| Risk | Mitigation |
|---|---|
| Composition schema too rigid for future brand-atoms extension | `extends_brand` is optional; doc notes the extension point |
| Prompt-conformance rendering is hand-waved | XAIP doc defines the rendering algorithm explicitly |
| Starter personas from feat/starter-persona-atoms use incompatible schema | New personas use composition-v1.json; old personas in separate dirs are not validated by this pipeline |
